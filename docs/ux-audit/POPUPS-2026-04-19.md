# Watt City — popup / overlay audit

**Date**: 2026-04-19
**Trigger**: user report that "some popup windows are disruptively made"
**Method**: read every fixed/sticky-positioned component, grep the live HTML for their presence, trace trigger conditions and dismiss behaviour, reason about stacking order on mobile.

## Inventory of live popups

| # | Component | Position | z-index | Triggered by |
|---|---|---|---|---|
| 1 | `OnboardingTour` | `fixed inset-0` (full-screen) | 50 | first login until `tourSeen` grants |
| 2 | `TeacherOnboardingTour` | `fixed inset-0` | 50 | teacher first-login |
| 3 | `PostGameBreakdown` | `fixed inset-0` | 50 | every `/api/score` success with `multBreakdown` |
| 4 | `CookieConsent` | `fixed bottom-0 inset-x-0` | 40 | no `wc_cookie_consent_v1` in localStorage |
| 5 | `TierUpToast` | `fixed bottom-24 left-1/2` | 40 | poll hits `pendingCelebration:true` |
| 6 | `NewGameToast` | `fixed bottom-5 left-1/2` | 40 | poll detects new AI game id |
| 7 | `CashflowHud` (mobile) | `fixed inset-x-2 bottom-[calc(3.5rem+sai)]` | 35 | every authed page |

All five overlay-class elements (CookieConsent, TierUpToast, NewGameToast, CashflowHud mobile, BottomTabs) compete for the bottom of the viewport on phones. That's the entire finding in one sentence.

## Concrete disruptive issues

### 1. CookieConsent covers BottomTabs on first load (CRITICAL)

- `components/cookie-consent.tsx:62` — `fixed bottom-0 inset-x-0 z-40`.
- `components/bottom-tabs.tsx:26` — `fixed bottom-0 inset-x-0 z-20`.
- Both claim the same viewport slot; CookieConsent wins (higher z). Until the user taps "Rozumiem", the primary mobile nav (`🏙️ Miasto · 🎮 Gry · 🛒 Giełda · 👤 Profil`) is **hidden behind the consent banner**.
- The banner is mounted from `app/layout.tsx:158` inside every logged-in + anonymous page — first-touch experience = a banner blocking navigation.
- Severity: **HIGH**. Fix: stack the banner `bottom-14` on `<sm` so it sits above BottomTabs instead of on top of them, or use a dismissible corner chip.

### 2. CashflowHud cannot be dismissed on mobile (HIGH)

- `components/cashflow-hud.tsx:178` — dismiss button is `className="hidden sm:inline-flex …"`. On phones (<640px) there's **no way to hide the HUD**. It sits at `bottom-[calc(3.5rem+env(safe-area-inset-bottom))]` above the BottomTabs, permanently.
- Paired with #3 below, this means the bottom 120-140px of every page is reserved for UI chrome. On iPhone SE (375×667), that's ~20% of the viewport.
- Fix: either show the `✕` on mobile too, or collapse the HUD to a single-line pill that the user can tap to expand.

### 3. CashflowHud dismissal doesn't persist across navigation (HIGH)

- `components/cashflow-hud.tsx:99` — `const [dismissed, setDismissed] = useState(false)`. No localStorage, no cookie, no server flag. Every page mount reinitialises `dismissed:false`.
- A desktop user who closes the HUD on `/miasto` sees it reappear when they click `/games`. The close button is therefore a per-page hide, not a preference — surprising behaviour.
- Exception: when `hud.alertLevel === "critical"` the component refuses to dismiss (line 121). That's sensible for a brownout cliff; the fix is only for non-critical state.
- Fix: persist dismissal in localStorage with an `alertReason` key so it auto-re-opens when the reason changes (already half-implemented at `:117-119`).

### 4. Four competing toasts/banners at bottom-center (HIGH)

On mobile the vertical ordering from bottom up is:
```
BottomTabs          0 → 56px   (z-20)
CookieConsent       0 → ~60px  (z-40, covers BottomTabs)
NewGameToast        20px → ~140px (z-40, overlaps CookieConsent + CashflowHud)
CashflowHud         56px → ~120px (z-35)
TierUpToast         96px → ~180px (z-40, overlaps CashflowHud)
```

- If `TierUpToast` + `CashflowHud` + `CookieConsent` + `BottomTabs` are all live at once, there's no room on a 667px tall viewport — they pile on one another. Live HTML confirms the mounts unconditionally from `app/layout.tsx`.
- The `NewGameToast` doesn't even account for BottomTabs — `bottom-5` on a mobile device drops the toast directly into the tab row and half of it gets covered.
- Fix: adopt a single `ToastStack` region (one `aside` at `bottom-[calc(3.5rem+sai+1rem)]` on mobile, one at `top-20 right-3` on desktop) and mount all toasts into it. One source of truth for vertical offsets; no individual component picks `bottom-5`.

### 5. PostGameBreakdown opens AFTER EVERY GAME (HIGH friction)

- `components/games/round-result.tsx:59-65` — `useEffect(…setBreakdownOpen(true))` fires whenever `multBreakdown !== null` on the score response. Server side, `v2_post_game_modal` is at 100% per D7, so `multBreakdown` is always returned.
- Consequence: every single game (9 evergreen + 12 AI kinds) closes with a full-screen `role="dialog"` `aria-modal="true"` blocking interaction until the user taps OK or Escape.
- The modal is educationally useful the first 2-3 times (explains the multiplier ladder). By play #20 it's pure friction.
- Fix options:
  - Collapse the modal into an inline "explain multipliers ▾" disclosure on the `RoundResult` card (no new dialog).
  - Gate the modal on `plays % 10 === 0` or `seenTimes < 3` (store on profile).
  - Allow "don't show again" checkbox that writes `onboarding.seenPostGameBreakdown = true`.

### 6. Backdrop-click dismisses PostGameBreakdown + Onboarding too eagerly (MEDIUM)

- `components/post-game-breakdown.tsx:97` — `onClick={onClose}` on the full-screen backdrop. A fat-fingered tap anywhere outside the 28rem card closes the dialog.
- Mobile users scrolling after a score routinely dismiss the modal accidentally. Combined with #5 (shown every game), most kids never read it.
- `components/onboarding-tour.tsx:108` does NOT have backdrop-click dismiss — only the explicit "Skip" link. Good pattern. Copy that over to PostGameBreakdown.
- Fix: remove the outer `onClick={onClose}`. Keep the explicit "OK" button + Escape key.

### 7. OnboardingTour blocks on every first-login tab (MEDIUM)

- `components/onboarding-tour.tsx:73-89` — mounts in `app/layout.tsx:199`, fetches `/api/me/profile`, shows full-screen dialog when `!tourSeen`.
- Opens on ANY first page after signup — `/`, `/miasto`, `/games`, any deep link. User who bookmarks `/games/ai/XYZ` gets the onboarding dialog instead of the game.
- The network fetch happens after the page hydrates, so there's a visible "flash of page content → modal pop" that on slower connections can reach 300-500ms.
- Fix: only mount `OnboardingTour` on `/` (guard in layout by `pathname === "/"` via `usePathname()`), and render a skeleton (fixed backdrop without the card) during the fetch so the flash is smooth.

### 8. NewGameToast fires globally every 30s (MEDIUM)

- `components/new-game-toast.tsx:61` — `setTimeout(poll, 30_000)` runs on every authed page. Polls `/api/events/latest-game` every 30s.
- If the user is mid-loan-form or mid-game and a new AI challenge drops, the toast pops at `bottom-5` (covers CookieConsent + BottomTabs) and `router.refresh()` forces a soft reload of server data. On a form page this resets unsaved inputs. Friction.
- Fix: gate on `pathname` — don't mount on `/games/ai/*`, `/loans/*`, `/register`, `/klasa/*`, `/nauczyciel/*`. Or at least don't `router.refresh()` when a form is in edit state (detect via `dirty` state in context).

### 9. TierUpToast uses `bottom-24` without accounting for CashflowHud (MEDIUM)

- Toast at `bottom-24` (96px) lands directly inside the CashflowHud band (56-120px on mobile). User gets a confetti banner ON TOP of their balance HUD. Visually they merge.
- Fix: raise to `bottom-[calc(3.5rem+5rem+sai)]` on `<sm` so it clears CashflowHud, or fold into the same ToastStack from #4.

### 10. WattDeficitPanel is a sticky banner with in-memory dismissal only (LOW)

- `components/watt-deficit-panel.tsx:150` — `sticky top-16 z-[30]` pushes all page content down whenever `deficitState.inDeficit` is true.
- `components/watt-deficit-panel.tsx:186` — dismiss is `useState(setDismissed)` — no persistence. Navigate away → panel returns. User cannot "acknowledge and continue."
- Educational intent is good (this is the brownout cliff from V3.3) but the implementation makes it feel like an error banner that won't go away.
- Fix: persist dismissal keyed on the `deficitState.startedAt` timestamp — panel re-opens if deficit restarts, stays dismissed while the same deficit continues.

### 11. NotificationBell dropdown overlaps nav items on narrow viewports (LOW)

- `components/notification-bell.tsx:92` — `absolute right-0 top-10 w-[min(95vw,22rem)]`. On 360px wide phones the dropdown is 95vw = 342px, which extends past the bell's left edge and sits under the user-chip + language-switcher. Tap outside the dropdown to close hits the language switcher instead.
- Fix: either `right-[-1rem]` on `<sm` or use a `portal` to render above everything.

## Suggested fixes — priority order

1. **#1 CookieConsent vs BottomTabs** — the critical one-touch issue. Ship as a dict-only + class-only patch (no component API change). Effort: 15 min.
2. **#5 PostGameBreakdown modal fatigue** — gate on "first 3 plays" or inline. Effort: 1-2 h.
3. **#4 / #9 bottom-stack** — one ToastStack component, migrate 3 callers. Effort: 2-3 h.
4. **#2 / #3 CashflowHud dismiss UX** — mobile button + localStorage persist. Effort: 30 min.
5. **#6 backdrop-click** — one line delete. Effort: 5 min.
6. **#7 OnboardingTour pathname gate** — three lines. Effort: 15 min.
7. **#8 NewGameToast pathname gate + form-safe refresh** — route guard + context. Effort: 1 h.
8. **#10 WattDeficitPanel persistent dismissal** — localStorage keyed on start timestamp. Effort: 30 min.
9. **#11 NotificationBell portal or offset** — 5 min.

Of these, (1), (5), (6) are the most visible "disruptive" ones a user would flag — and all three are quick to fix.

## Appendix — evidence

- Layout mount order: `app/layout.tsx:152-205` — `CookieConsent` + `CashflowHud` + `NewGameToast` + `OnboardingTour` + `TierUpToast` + `WattDeficitPanel` all inside the same `<body>` layout subtree, so they co-exist on every authed page.
- BottomTabs z-index: `components/bottom-tabs.tsx:26` — `z-20`.
- CookieConsent z-index: `components/cookie-consent.tsx:62` — `z-40`.
- Live HTML on `/` and `/miasto` + `/games` show `z-20` (nav), `z-[35]` (HUD) as the only initial-render fixed elements; everything else is client-mounted conditionally.
