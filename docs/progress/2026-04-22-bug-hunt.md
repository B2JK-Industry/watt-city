# 2026-04-22 — post-UX-fix bug hunt + hardening sweep

After the UX-fix batch landed (commit `f124349`, see
`2026-04-22-ux-fixes-batch.md`), a two-agent audit surfaced four
regressions introduced by that commit plus seven pre-existing bugs
across data-integrity, security, and perf. All eleven addressed in
the present commit.

## Regressions from `f124349` (my commit, fixed here)

**R1 — `/api/score` fire-and-forget lost on Vercel.** `void backgroundWork`
does not keep the promise alive after `Response.json()` flushes; Vercel
may terminate the invocation with `recordEvent` / `sweepAchievements`
mid-flight, silently losing analytics + delaying achievements by one
poll cycle. Fixed by switching to Next.js `after()` from `next/server`
(stable since 15.1, available in our 16.x). `after()` participates in
the runtime's lifetime tracking so the bg work always completes.

**R2 — notification-bell `role="dialog"` without focus trap.** ARIA
`role="dialog"` promises a modal dialog contract (focus trap, Escape
dismissal, aria-modal, focus-return on close). The popover provided
none of those — NVDA/JAWS announce a dialog that keyboard users can't
leave. Fixed by dropping to `role="region"` (accurate for a labelled
notification panel) and adding an explicit Escape-key listener so both
mouse and keyboard paths dismiss.

**R3 — `/api/admin/purge-e2e-accounts` too aggressive.** The default
prefix list included `kid`, `bot`, `sec`, `db`, `di`, `okuser` — all
names a real user could plausibly register. `hardErase` also burns
web3 medals (GDPR Art. 17) so a false positive is unrecoverable.
Two-part fix:
- Default prefix list narrowed to multi-char, unambiguous idioms only
  (`gp`, `pr`, `rl`, `ghost`, `smoke`). Ambiguous prefixes move to a
  new `includeAmbiguous: true` opt-in.
- Added a `confirm: "purge-e2e-accounts-for-real"` magic-token gate;
  `dryRun: false` without the token returns 400 with a hint. Belt and
  braces against an operator typing "no" by reflex.
- Each per-user `hardErase` failure now logs to stderr so Vercel logs
  capture enough context to resume a partial wipe.
- `scripts/purge-e2e-accounts.sh` learned `--include-ambiguous`
  and auto-injects the confirm token on `--commit`.

**R4 — onboarding-tour `wc:open-tour` could reset mid-flow.** A stray
dispatch while the tour was already open snapped the user back to
step 0. Fixed by wrapping the reopen handler in `setNeedsTour((cur) =>
cur ? cur : …)` so an already-open tour is a no-op.

## Pre-existing bugs fixed here

**B1 — `getPlayerState → mutate → savePlayerState` is not atomic.**
The `PlayerState` JSON blob at `xp:player:<u>` is read, mutated in
memory, then written back. Two concurrent mutations (two tabs, a
double-click, a buy racing with a build) both read the same state,
both apply their mutation, the second `savePlayerState` overwrites
the first. `awardXP` had the right pattern via a single-flight lock
in `lib/leaderboard.ts`; every other mutator was unguarded.

Fix: new `lib/player-lock.ts` exports `withPlayerLock(username, body)`
and `withListingLock(listingId, body)` — generalised from the
`withAwardLock` template (5 s TTL, 5× exp-backoff, best-effort
release). Applied across every read-modify-write mutator:

- `/api/buildings/{place, upgrade, demolish}` — all wrap the critical
  section. `/api/buildings/tick` already has its own tick-lock inside
  `tickPlayer`; no new wrap needed.
- `/api/loans/{take, take-generic, repay-extra, rescue, bankructwo,
  [id]/auto-repay}` — wrapped.
- `/api/market/list` — wrapped with seller's player lock.
- `/api/market/buy/[id]` — wrapped with outer `withListingLock(id)`
  plus inner `withPlayerLock(buyer)`. Re-checks `listing.status ===
  "active"` after acquisition (double-checked locking). Closes B2.
- `/api/market/cancel/[id]` — wrapped with listing lock.
- `/api/me/profile` PATCH, `/api/me/notifications` POST, `/api/me/tier`
  POST, `/api/parent/child/[username]/web3-consent` — wrapped.

**B2 — marketplace buyListing had no listing lock.** Two buyers on
the same listing could both pass the `status === "active"` guard,
each get the building, each get charged, seller credited only once.
Closed as part of B1 — `/api/market/buy` now holds the listing lock
across buyer + seller state writes, plus double-checks the status
after acquisition.

**B3 — admin + cron bearer compare was not constant-time.** `token ===
secret` short-circuits on the first mismatched char — theoretically
gives a network-timing attacker a channel to probe `CRON_SECRET` /
`ADMIN_SECRET` byte-by-byte. `/api/cron/*` is session-less + CSRF-
exempt (the most exposed surface), so this is worth closing. Fixed
in `lib/admin.ts` + `lib/cron-auth.ts` with `crypto.timingSafeEqual`
on length-equal Buffers (length mismatch short-circuits to false
before `timingSafeEqual` is called; no secret-length leak beyond
what a single query teaches).

**B4 — `/api/analytics/web-vitals` unauthenticated + unbounded keyspace.**
Endpoint is CSRF-exempt (sendBeacon can't set headers) and has no
rate limit. `body.route` accepted up to 120 arbitrary chars; the
`hi:` ZSET member was keyed by `Math.floor(body.value)` creating up
to 60 001 unique members per (day, route, metric) tuple. Attacker fills
the free-tier Upstash keyspace in minutes. Fix:
- `route` normalised to a known-route enum (`KNOWN_ROUTES`); unknown
  routes collapse to `"other"` or a `/{segment}/:slug` prefix for the
  few dynamic families (`/games/:slug`, `/parent/:slug`, …).
- `hi:` buckets quantised to 50 ms — max 1 201 members per tuple.
- Per-IP rate limit 120 req/min via `rateLimit("web-vitals:<ip>", …)`.

**B5 — Login IP rate limit incompatible with school NAT.** 25 kids
behind a single classroom NAT exit IP could trip `LOGIN_IP_LIMIT=20/min`
collectively. Fix:
- Per-IP default raised to 60/min (still DoS-expensive for a stuffer
  at ~86 k/day).
- New per-username limit (`LOGIN_USER_LIMIT=10 / 15 min`) catches an
  attacker who rotates IPs — 10 password failures on one username in
  15 minutes triggers a 429 regardless of source IP.

**B6 — `rateLimit` GET-then-SET race.** Two parallel callers could
both read `current=0`, both write `1`, both pass a limit check — an
attacker could burst through the nominal cap under load. Fix:
- New `kvIncrWithTTL(key, ttlSeconds)` in `lib/redis.ts` uses Upstash
  `MULTI → INCR → EXPIRE → EXEC` (single round-trip, server-atomic).
  Memory fallback is synchronous map increment (JS event loop keeps
  it race-safe).
- `lib/rate-limit.ts` rewritten to INCR then compare. Counter
  increments even on reject, which is intentional — a caller spamming
  the endpoint at 100 rps gets every request counted so the window
  extension anti-grief actually bites.

**B7 — `/api/me/notifications` POST set-prefs had no Zod validation.**
A user could persist garbage like `{quietHours: {start: "🔥"}}`
into their own prefs, self-DoS-ing the next notification-panel
render. Fix:
- Added discriminated-union `BodySchema` for `action: "mark-seen"`
  vs `action: "set-prefs"` (with nested `PrefsSchema`).
- `writePrefs` upgraded from shallow to deep merge so a partial
  `{quietHours: {start: 22}}` patch doesn't wipe siblings.
